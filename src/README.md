# Шаг 2: централизованный список сохраняемых полей

Новый файл **`game/saveableFields.js`** — единственный источник правды по всем
48 полям игрока, которые сохраняются в Supabase. Раньше этот список был
вручную продублирован в 4 местах, и любое забытое поле означало тихую
поломку без единой ошибки в консоли:

1. `stateRef.current = {...}` — сборка снапшота из state (осталась как была,
   технически неизбежна: React setState-переменные нельзя перечислить в цикле)
2. `buildSnapshot()` — маппинг camelCase → имя колонки в БД → **теперь генерируется**
3. Хидратация в `selectNickname()` при логине (48 строк `if (row.x != null) setX(...)`) → **теперь генерируется**
4. `dbLoadGuildMembers` — select-список для вкладки "Гильдия" → **теперь генерируется**

Теперь, чтобы добавить новое сохраняемое поле, нужно:
1. `useState` в `AppInner` (как и раньше)
2. одна строка в `stateRef.current = {...}` (как и раньше, там же)
3. одна строка в `fieldSetters` (сразу под stateRef)
4. одна строка в `game/saveableFields.js` — `{ key, column, guildVisible }`

`buildSnapshot`, хидратация при логине и select для гильдии обновятся сами —
их больше не нужно трогать руками.

## Проверка (не только сборка — поведение)
Раз этот шаг меняет реальную логику (не просто перекладывает код), одной сборки
недостаточно — я явно протестировал, что новый код **на тех же входных данных
выдаёт тот же результат**, что и старый:
- `buildSnapshot()`: тест на фейковом `stateRef.current` с 48 уникальными
  значениями — старая ручная логика и новая генерируемая дают **идентичный** объект.
- Хидратация при логине: тест на фейковой строке из БД (включая `null`/`undefined`
  поля, чтобы проверить, что "не трогать несуществующее поле" тоже работает
  одинаково) — набор вызовов сеттеров **идентичен** старому поведению.
- Кросс-сверка всех 4 старых списков (`stateRef`, `buildSnapshot`, хидратация,
  select гильдии) против `SAVEABLE_FIELDS` — совпадение 1-в-1 по всем 48 полям.
- esbuild-бандл — 0 ошибок, 0 предупреждений.

## Находка по пути (не исправлял, только зафиксировал)
При сверке обнаружилось, что поле `active_shield` (активный щит игрока) сейчас
**не входит** в select для вкладки "Гильдия" — то есть щит не виден другим
игрокам в профиле, хотя логически было бы естественно его показывать. Похоже
на недосмотр, а не намеренное решение. Оставил `guildVisible: false` для этого
поля, чтобы шаг 2 ничего не менял в видимом поведении — но если это баг,
исправление теперь буквально одна строка (`false` → `true`) в
`game/saveableFields.js`. Скажи — включу.
